# Common Errors and Solutions

This guide covers error messages you may encounter when connecting to and working with databases through WhoDB.

## Connection Errors

<AccordionGroup>
<Accordion title="Connection refused">

### Error Message
```
Connection refused
Unable to connect to the database
```

### What It Means
WhoDB tried to connect to the database server, but the connection was rejected. This usually means the database server isn't running, isn't listening on the specified port, or isn't reachable from where WhoDB runs.

### Solutions

**1. Check connection parameters in the login form:**
- Host: Use the correct hostname or IP
- Port: Verify the correct port (PostgreSQL: 5432, MySQL: 3306, MongoDB: 27017)
- Database name and credentials

**2. Docker users — check your host value:**
If WhoDB runs in Docker and the database runs on your host machine, `localhost` refers to the WhoDB container itself. Use `host.docker.internal` (macOS/Windows) or a shared Docker network instead. See the [Docker integration guide](/resources/integrations/docker).

**3. Verify the database is reachable from the WhoDB host:**
```bash
nc -zv hostname 5432
```

If this fails, the problem is network reachability or database availability, not WhoDB. Start the database service or fix the network path first.

See [Connection Issues Troubleshooting](/resources/troubleshooting#connection-issues) for detailed steps.

</Accordion>

<Accordion title="Connection timeout">

### Error Message
```
Connection timeout
Connection attempt timed out
```

### What It Means
WhoDB reached out to the server but didn't receive a response within the timeout period. The network is slow, a firewall is dropping packets, or the database is unresponsive.

### Solutions

**1. Test with the native client first:**
Use `psql`, `mysql`, `mongosh`, or the equivalent client against the same host, port, and credentials. If the native client also times out, the problem is network reachability, database health, or an upstream timeout rather than a configurable WhoDB web setting.

**2. Reduce query load if the timeout happens mid-session:**
- Add WHERE conditions to filter data
- Use LIMIT to reduce results
- Lower the page size in [Settings](/features/settings)

**3. Check database or proxy timeouts:**
If the same query dies outside WhoDB, review the timeout that actually applies in your stack, such as PostgreSQL `statement_timeout` or a reverse proxy read timeout.

</Accordion>

<Accordion title="Authentication failed / Access denied">

### Error Message
```
Authentication failed
Access denied for user 'username'@'hostname'
Invalid username or password
```

### What It Means
The database server rejected the credentials entered in the WhoDB login form. Either the username doesn't exist, the password is wrong, or the user isn't allowed to connect from the host WhoDB runs on.

### Solutions

**1. Verify credentials outside WhoDB:**
```bash
# PostgreSQL
psql -h hostname -U username -d database

# MySQL
mysql -h hostname -u username -p
```

If the native client connects with the same values, re-enter them carefully in the login form — passwords are case-sensitive and leading/trailing spaces count.

**2. Check for special characters in the password:**
Enter the password exactly as-is in the login form; WhoDB does not require URL-encoding form fields. If a native client works but WhoDB doesn't, test temporarily with a simpler password to rule out an entry mistake.

**3. Check host-based access rules:**
Some databases restrict which hosts a user may connect from (`pg_hba.conf`, MySQL `'user'@'host'` grants). Remember that when WhoDB runs in Docker, the connection originates from the container's address, not your workstation's.

**4. Environment-defined profiles:**
If you connect through a profile defined with environment variables such as `WHODB_POSTGRES_1`, verify the JSON contains the right credentials and restart WhoDB after changing it.

</Accordion>

<Accordion title="SSL certificate problem: self signed certificate">

### Error Message
```
SSL certificate problem: self signed certificate
Certificate verification failed
```

### What It Means
The database presents a self-signed SSL certificate that can't be verified against a trusted CA. This is common in development and testing environments.

### Solutions

**1. Lower the SSL verification mode:**
In the login form's advanced panel, change `SSL Mode` from `verify-full` to `require`. This encrypts the connection without verifying the certificate chain.

**2. Provide the CA certificate:**
For `verify-ca` or `verify-full`, paste the PEM CA certificate into the `SSL CA Content` field in the advanced panel. Get the PEM from your database provider or export it:
```bash
openssl s_client -connect hostname:5432 -starttls postgres -showcerts
```

**3. Disable SSL (local development only):**
Set `SSL Mode` to `disable`. Never do this in production.

</Accordion>

<Accordion title="SSL: certificate hostname mismatch">

### Error Message
```
SSL: CERTIFICATE_VERIFY_FAILED hostname mismatch
The hostname doesn't match the certificate
```

### What It Means
The SSL certificate is valid but issued for a different hostname than the one in the login form — for example, the certificate is for `db.example.com` but you're connecting to an IP address.

### Solutions

**1. Connect using the hostname on the certificate** instead of an IP address.

**2. Set the `SSL Server Name` field** in the advanced panel to the hostname the certificate was issued for, while keeping the host/IP you actually connect to.

**3. Use `SSL Mode: verify-ca`** instead of `verify-full` — this verifies the certificate chain without checking the hostname.

</Accordion>
</AccordionGroup>

## Session Errors

<AccordionGroup>
<Accordion title="Logged out unexpectedly / session no longer valid">

### What It Means
In the web app, your database credentials are stored encrypted server-side and tied to an `HttpOnly` session cookie. If the cookie is missing, blocked, or the server-side session is gone, WhoDB returns you to the login page.

### Causes
- WhoDB server or container was restarted without persistent session storage
- Browser or extension is blocking cookies for the WhoDB origin
- A reverse proxy is stripping or rewriting the session cookie
- You logged out in another tab (logout deletes the server-side session)

### Solutions

**1. Log in again** — saved connection profiles in your browser keep the non-secret details (host, database, connection name), so you only need to re-enter the password.

**2. Persist session data in Docker:**
The image writes session data to the `/data` volume. Mount it so sessions survive container restarts:
```bash
docker run -it -p 8080:8080 -v whodb-data:/data clidey/whodb
```

**3. Check cookie handling:**
- Allow cookies for the WhoDB origin in your browser
- If behind a reverse proxy, ensure it forwards cookies unmodified

</Accordion>

<Accordion title="Sessions fail when running as a non-root user">

### What It Means
The container writes session data to the `/data` volume. When you run the container with `--user`, that UID must be able to write to the volume, or logins fail.

### Solution
Make sure the mounted volume is owned by (or writable for) the UID you run as:
```bash
docker run --user 1000:1000 -v whodb-data:/data clidey/whodb
# Ensure the volume is writable by UID 1000
```

</Accordion>
</AccordionGroup>

## Import and Export Errors

<AccordionGroup>
<Accordion title="Import file rejected (too large or too many rows)">

### What It Means
Data import enforces limits per file:

- Maximum file size: `50 MB`
- Maximum row count: `100,000`

Files beyond either limit are rejected before import starts.

### Solutions

**1. Split the file** into chunks under 50 MB and 100,000 rows each, and import them sequentially.

**2. Use database-native bulk tooling for very large loads** (`COPY`, `LOAD DATA INFILE`, `mongoimport`, and so on) — WhoDB import is designed for small-to-medium datasets.

See [Importing Data](/data/importing) for supported formats and column-mapping details.

</Accordion>

<Accordion title="Import fails partway with a database error">

### What It Means
The file passed WhoDB's checks, but the database rejected one or more rows — for example a constraint violation or a value that doesn't match the column type. The error message shown comes from your database.

### Solutions

**1. Read the reported row/column** in the error and fix the source file (missing required values, duplicate keys, wrong date formats are the usual suspects).

**2. Verify the column mapping** matches the table's schema before re-running the import.

</Accordion>

<Accordion title="Export runs out of memory or is very slow">

### What It Means
Exporting a very large result set in one pass can exhaust memory or take a long time.

### Solutions

**1. Filter before exporting** — apply WHERE conditions or a query so you export only the rows you need.

**2. Export in batches** — split by date range or key range across multiple exports instead of one huge file.

**3. Use database-native tools** (`pg_dump`, `mysqldump`, etc.) for full backups or migration-scale dumps. See [Export Options](/advanced/export-options).

</Accordion>
</AccordionGroup>

## Docker and Deployment Errors

<AccordionGroup>
<Accordion title="Can't reach a localhost database from the WhoDB container">

### What It Means
Inside a container, `localhost` is the container itself — not your host machine.

### Solutions

**macOS/Windows:** use `host.docker.internal` as the host in the login form.

**Linux:** run with host networking, or put both containers on a shared network:
```bash
docker run -it -p 8080:8080 --network host clidey/whodb
```

**Docker Compose:** put WhoDB and the database on the same network and use the service name as the host. See the [Docker integration guide](/resources/integrations/docker).

</Accordion>

<Accordion title="Port 8080 already in use">

### Error Message
```
bind: address already in use
```

### Solutions

**1. Map a different host port:**
```bash
docker run -it -p 3000:8080 clidey/whodb
```

**2. Or find and stop the conflicting process:**
```bash
lsof -i :8080
```

</Accordion>

<Accordion title="Container healthcheck failing">

### What It Means
WhoDB CE exposes `GET /health`, which returns plain text `ok`. If your orchestrator reports the container unhealthy, the server isn't responding on the expected port.

### Solutions

**1. Use a healthcheck that matches the image** (the image includes `wget`):
```yaml
healthcheck:
  test: ["CMD-SHELL", "wget -qO- http://localhost:8080/health | grep -qx ok"]
```

**2. Check container logs** for startup errors:
```bash
docker logs whodb
```

See [Monitoring Integration](/resources/integrations/monitoring) for full examples.

</Accordion>
</AccordionGroup>

## Resource Errors

<AccordionGroup>
<Accordion title="Out of memory">

### What It Means
An operation — usually a huge result set, export, or mock-data generation — required more RAM than available.

### Solutions

**1. Reduce result set size:** add WHERE conditions and LIMIT, and lower the page size in [Settings](/features/settings).

**2. Export in smaller batches** by date range or specific columns.

**3. Generate mock data in smaller runs** (for example 1,000 rows at a time).

</Accordion>

<Accordion title="Connection pool exhausted / too many connections">

### What It Means
Either WhoDB's internal pool or the database server's connection limit is saturated.

The current CE server keeps an internal SQL connection pool with conservative defaults (max open: `10`, max idle: `5`, max lifetime: `30 minutes`, max idle lifetime: `5 minutes`). These are fixed defaults in the server.

### Solutions

**1. Reduce concurrency:** avoid running many heavy queries simultaneously; wait for operations to finish.

**2. Check the database server's own limit** (`max_connections` or equivalent) — other applications sharing the database count against it.

**3. Add a database-side pooling layer** such as PgBouncer or ProxySQL when many services share one database.

**4. Restart WhoDB** to reset the pool if connections appear stuck.

</Accordion>
</AccordionGroup>

## SQL Errors from Your Database

Errors such as syntax errors, unknown tables or columns, constraint violations, type mismatches, and division by zero are produced by your database, not by WhoDB — WhoDB surfaces them verbatim. Consult your database's own documentation (PostgreSQL, MySQL, SQLite, etc.) for the exact meaning, and see [Query Optimization](/best-practices/query-optimization) for guidance on writing efficient queries.
