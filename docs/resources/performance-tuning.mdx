# Performance Tuning Guide

Learn how to get the best performance out of WhoDB when working with large datasets. This guide covers WhoDB's page size settings, connection pooling behavior, caching, and practical advice for queries and exports.

For general SQL tuning — indexing, EXPLAIN plans, and query rewriting — see [Query Optimization](/best-practices/query-optimization) and your database's official documentation.

## Page Size and Pagination

<AccordionGroup>
<Accordion title="Tuning page size">

### How Pagination Works

WhoDB paginates table views and query results automatically. Two settings control it:

- **Default page size** — set in [Settings](/features/settings); applies to new table views
- **`WHODB_MAX_PAGE_SIZE`** — server-side environment variable capping the maximum rows per page request (default `10000`)

The settings page controls the default within the limit the backend enforces. See [Sorting and Pagination](/data/sorting-pagination) for the UI controls.

### Practical Guidance

- **Lower the page size** (100–500) when working with wide tables, large text/JSON columns, or a slow network — less data per request keeps the UI responsive
- **Raise the page size** only when you genuinely need to scan more rows at once and the rows are small
- If you need more than `WHODB_MAX_PAGE_SIZE` rows in one request, raise the env var and restart WhoDB:

```yaml
environment:
  - WHODB_MAX_PAGE_SIZE=20000
```

Prefer filters over huge pages — a filtered 100-row page is faster than an unfiltered 10,000-row page in every respect.

</Accordion>
</AccordionGroup>

## Connection Pool

<AccordionGroup>
<Accordion title="Connection management in the current CE build">

The current CE server keeps an internal SQL connection pool with conservative defaults:

- Max open connections: `10`
- Max idle connections: `5`
- Max connection lifetime: `30 minutes`
- Max idle lifetime: `5 minutes`

These values are fixed defaults in the server.

### Practical Tuning Guidance

- Keep default WhoDB usage focused on filtered reads, not full-table scans
- Lower result sizes and page sizes before assuming the pool is the bottleneck
- Raise database-side connection limits only when the database server is the actual constraint
- Use a database-side pooling layer such as PgBouncer or ProxySQL when multiple apps share the same database

### Database-Side Connection Limits

WhoDB's pool counts against your database server's own limit, alongside every other client:

```sql
-- PostgreSQL
SHOW max_connections;

-- MySQL
SHOW VARIABLES LIKE 'max_connections';
```

Monitor actual usage on the database server and don't assume WhoDB is the consumer — an application connection leak elsewhere produces the same symptoms.

</Accordion>
</AccordionGroup>

## Scratchpad Query Tips

<AccordionGroup>
<Accordion title="Keeping Scratchpad queries fast">

### Use LIMIT and Filters

```sql
-- Unbounded: may transfer millions of rows to the browser
SELECT * FROM orders;

-- Bounded: fast and predictable
SELECT * FROM orders
WHERE created_at > '2024-01-01'
LIMIT 100;
```

### Select Only Needed Columns

```sql
-- Transfers all columns, including large JSON/text
SELECT * FROM products;

-- Transfers only what you need
SELECT id, name, price FROM products;
```

### Aggregate in the Database

```sql
-- Let the database do the work instead of pulling rows out
SELECT COUNT(*) FROM orders;
SELECT AVG(amount) FROM orders WHERE user_id = 5;
```

### Diagnose Slow Queries

If a filtered, limited query is still slow, the fix is database-side — indexes and execution plans. Run `EXPLAIN` in a Scratchpad cell and see [Query Optimization](/best-practices/query-optimization), or consult your database's documentation.

</Accordion>
</AccordionGroup>

## Exporting Large Datasets

<AccordionGroup>
<Accordion title="Export batching strategies">

### In WhoDB

1. Use filters to reduce the result set before exporting
2. Export in smaller batches if memory is limited — split by date range or key range:

```sql
SELECT * FROM orders WHERE created_at BETWEEN '2024-01-01' AND '2024-01-31';
SELECT * FROM orders WHERE created_at BETWEEN '2024-02-01' AND '2024-02-28';
```

3. Export only needed columns rather than `SELECT *`
4. CSV is usually faster and smaller than other formats for large data
5. Multiple exports with different filters beat one huge export

### When to Use Native Tools

For full backups, restores, or migration-scale dumps, use your database's native tooling (`pg_dump`, `mysqldump`, `COPY`, and so on) — WhoDB exports are designed for small-to-medium result sets. See [Export Options](/advanced/export-options).

</Accordion>
</AccordionGroup>

## Caching

<AccordionGroup>
<Accordion title="What WhoDB caches">

The frontend caches GraphQL responses (schema metadata, table lists, column definitions) in memory for the duration of your browser session, so repeat navigation does not re-query the database. Refreshing the page or using the refresh action re-fetches everything from the database — use that if the UI looks stale after a schema change.

</Accordion>
</AccordionGroup>

## Memory Usage

<AccordionGroup>
<Accordion title="Keeping memory usage predictable">

- Lower the default page size in [Settings](/features/settings)
- Use `LIMIT`, filters, and column selection in the Scratchpad
- Export large datasets in batches rather than in one pass
- Generate mock data in smaller runs (for example 1,000 rows at a time)
- Prefer database-native backup and bulk-export tools for very large jobs

Monitor the container while working:

```bash
docker stats whodb
```

</Accordion>
</AccordionGroup>

## Quick Checklist

- [ ] Page size suits your data width (lower for wide rows, large text/JSON)
- [ ] `WHODB_MAX_PAGE_SIZE` raised only if you actually need bigger pages
- [ ] Queries use WHERE, LIMIT, and explicit column lists
- [ ] Large exports split into filtered batches
- [ ] Database server's `max_connections` accounts for all clients, not just WhoDB
- [ ] Slow queries investigated with EXPLAIN and indexes on the database side
