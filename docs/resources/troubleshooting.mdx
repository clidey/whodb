# Troubleshooting Guide

This guide helps you resolve common issues when using WhoDB. Follow the step-by-step solutions to diagnose and fix problems.

## Connection Issues

### First: verify the database is reachable

Before adjusting anything in WhoDB, confirm the database is reachable from the machine (or container) WhoDB runs on:

```bash
# Basic reachability
nc -zv hostname 5432

# Full credential check with the native client
psql -h hostname -U username -d database    # PostgreSQL
mysql -h hostname -u username -p            # MySQL
mongosh --host hostname:27017 -u username -p # MongoDB
```

If the native client can't connect either, the problem is network reachability, database health, firewall rules, or credentials — fix that first. If the native client works but WhoDB doesn't, continue below.

<AccordionGroup>
<Accordion title="Connection refused or connection timeout">

### Symptoms
- Error: "Connection refused"
- Error: "Connection timeout"
- WhoDB displays "Unable to connect" message

### Step-by-Step Solution

<Steps>
<Step title="Verify connection parameters in the login form">
Double-check every field:
- **Host**: Correct hostname or IP address
- **Port**: Default ports (PostgreSQL: 5432, MySQL: 3306, MongoDB: 27017)
- **Database name**: Correct for SQL databases
- **Username/Password**: Correct credentials

**Docker users**: If running WhoDB in Docker and connecting to a database on your host machine, use `host.docker.internal` instead of `localhost` — inside the container, `localhost` is the container itself. See [Docker Networking](#docker-networking) below.
</Step>

<Step title="Verify the database is reachable">
Run the reachability checks at the top of this section from the host (or container) where WhoDB runs — not just from your workstation.
</Step>

<Step title="Check for slow or dropped connections">
If the connection hangs rather than failing fast, the same host/port test with the native client tells you whether it's a network or database issue rather than a configurable WhoDB web setting.
</Step>
</Steps>

</Accordion>

<Accordion title="SSL/TLS connection errors">

### Symptoms
- Error: "SSL certificate problem"
- Error: "SSL handshake failure"
- Connection works without SSL but fails with SSL enabled

### Step-by-Step Solution

<Steps>
<Step title="Choose the right SSL mode">
In the login form's advanced panel, `SSL Mode` controls verification:
- **disable**: No SSL (local development only)
- **require**: SSL required, certificate not verified
- **verify-ca**: SSL required, CA verified
- **verify-full**: SSL required, CA and hostname verified

Start with "require" if unsure.
</Step>

<Step title="Provide certificate content in the advanced panel">
For "verify-ca" or "verify-full", paste or upload the PEM content into the shared SSL fields:

- `SSL CA Content` — the CA certificate
- `SSL Client Cert Content` — client certificate for mutual TLS
- `SSL Client Key Content` — client private key for mutual TLS
- `SSL Server Name` — override the hostname used for verification

These fields take the certificate *content*, not a file path on the server. Download the PEM from your database provider and paste it in. (Some database types also expose path-based fields for files on the WhoDB host — see [Database Connectivity](/features/database-connectivity).)
</Step>

<Step title="Handle hostname mismatches">
If the error mentions a hostname mismatch:
- Connect using the hostname on the certificate instead of an IP, or
- Set `SSL Server Name` to the certificate's hostname, or
- Use `verify-ca` instead of `verify-full`
</Step>

<Step title="Inspect the certificate if needed">
```bash
openssl s_client -connect hostname:5432 -starttls postgres
```

This shows certificate details and any validation issues.
</Step>
</Steps>

</Accordion>

<Accordion title="Authentication failed">

### Symptoms
- Error: "Authentication failed"
- Error: "Access denied for user"
- Error: "Invalid credentials"

### Step-by-Step Solution

<Steps>
<Step title="Verify credentials outside WhoDB">
Test the same host, port, user, and password with the native client (see the reachability section above). This confirms whether the credentials themselves are correct.
</Step>

<Step title="Re-enter the password carefully">
Passwords are case-sensitive; watch for extra spaces, Caps Lock, and autofill surprises. Enter special characters exactly as-is — the login form does not require escaping or URL-encoding.
</Step>

<Step title="Check host-based access rules">
Databases can restrict which hosts a user connects from (`pg_hba.conf`, MySQL `'user'@'host'` grants). When WhoDB runs in Docker, the connection originates from the container's network address, which may not match a grant written for your workstation.
</Step>

<Step title="Check environment-defined profiles">
If you use profiles like `WHODB_POSTGRES_1`, verify the JSON credentials and restart WhoDB after changes.
</Step>
</Steps>

</Accordion>

<Accordion title="SSH tunnel or bastion connection fails">

### Symptoms
- Database only reachable via a bastion or private network
- Tunnel-based connection doesn't work

### Step-by-Step Solution

<Steps>
<Step title="Confirm where the tunnel should live">
The web app does not create SSH tunnels from the login form. Use one of these approaches:

- Create an external SSH tunnel yourself and point WhoDB at the forwarded host and port
- Deploy WhoDB where the database is already reachable over a private network
- Use the CLI if you want built-in SSH tunnel management
</Step>

<Step title="Create the tunnel manually">
```bash
ssh -L 5432:database-host:5432 username@ssh-host
```
</Step>

<Step title="Point WhoDB at the forwarded address">
Connect WhoDB to the forwarded endpoint:
- Host: `localhost` (or the host where your forward terminates)
- Port: the forwarded local port, such as `5432`

From WhoDB's perspective, it only needs the final reachable host and port. If WhoDB runs in Docker, remember the tunnel must be reachable from the container (`host.docker.internal` for a tunnel on your host machine).
</Step>
</Steps>

</Accordion>
</AccordionGroup>

## Session and Login State Issues

<AccordionGroup>
<Accordion title="Logged out unexpectedly or asked to log in again">

### Symptoms
- Returned to the login page mid-session
- "Session expired" or unauthorized errors

### How sessions work
In the web app, a successful login is stored encrypted server-side in a local session database; your browser only holds an opaque, `HttpOnly` session cookie. Credentials never reach browser storage — only non-secret display details (connection name, host, database) are kept there for the profile switcher.

### Step-by-Step Solution

<Steps>
<Step title="Persist session data across restarts">
The Docker image writes session data to the `/data` volume. Without a mounted volume, restarting the container invalidates all sessions:
```bash
docker run -it -p 8080:8080 -v whodb-data:/data clidey/whodb
```
</Step>

<Step title="Check cookie handling">
- Allow cookies for the WhoDB origin in your browser
- Privacy extensions and strict cookie policies can block the session cookie
- If WhoDB is behind a reverse proxy, ensure the proxy forwards cookies unmodified
</Step>

<Step title="Check volume permissions when running as non-root">
When running the container with `--user`, the `/data` volume must be writable by that UID, or logins and sessions fail.
</Step>

<Step title="Log in again">
Saved connection profiles keep non-secret details, so you only re-enter the password. Logging out in any tab deletes the server-side session for all tabs.
</Step>
</Steps>

</Accordion>
</AccordionGroup>

## Docker Networking

<AccordionGroup>
<Accordion title="Can't reach databases from the WhoDB container">

### Symptoms
- Database on your host machine is unreachable with `localhost`
- Database in another container is unreachable

### Step-by-Step Solution

<Steps>
<Step title="Database on the host machine (macOS/Windows)">
Use `host.docker.internal` as the host in the login form.
</Step>

<Step title="Database on the host machine (Linux)">
Use host networking:
```bash
docker run -it -p 8080:8080 --network host clidey/whodb
```
</Step>

<Step title="Database in another container">
Put both containers on the same Docker network and use the service name as the host:
```yaml
services:
  postgres:
    image: postgres:15
    networks: [whodb-network]
  whodb:
    image: clidey/whodb
    networks: [whodb-network]
networks:
  whodb-network:
```

Then connect with `Host: postgres`, `Port: 5432`.
</Step>

<Step title="Verify from inside the container">
```bash
docker exec -it whodb sh -c "nc -zv host.docker.internal 5432"
```

If this fails, the issue is the container's network path, not WhoDB.
</Step>
</Steps>

See the [Docker integration guide](/resources/integrations/docker) for complete examples.

</Accordion>
</AccordionGroup>

## Performance Issues

<AccordionGroup>
<Accordion title="Queries or table views are slow">

### Symptoms
- Queries take a long time to return results
- Table with millions of rows loads very slowly

### Step-by-Step Solution

<Steps>
<Step title="Check result size first">
```sql
SELECT * FROM large_table LIMIT 100;
```

If LIMIT 100 is fast but the full query is slow, the problem is result size — add filters or reduce the page size in [Settings](/features/settings).
</Step>

<Step title="Filter and reduce columns">
Add WHERE conditions and select only needed columns rather than loading whole tables.
</Step>

<Step title="Compare against the native client">
If the same query is equally slow in `psql`/`mysql`, the fix is database-side (indexes, execution plan) — see [Query Optimization](/best-practices/query-optimization).
</Step>
</Steps>

</Accordion>

<Accordion title="High memory usage">

### Symptoms
- WhoDB process consuming excessive RAM
- "Out of memory" errors

### Step-by-Step Solution

<Steps>
<Step title="Identify the memory-intensive operation">
Large SELECT results, big exports, and mock-data generation are the usual causes.
</Step>

<Step title="Reduce working set">
- Lower the page size and add LIMIT/filters for large queries
- Export in smaller batches (date ranges, fewer columns)
- Generate mock data in smaller runs
</Step>

<Step title="Monitor the container">
```bash
docker stats whodb
```

If memory keeps growing without activity, restart WhoDB and report an issue.
</Step>
</Steps>

</Accordion>

<Accordion title="UI is sluggish overall">

### Step-by-Step Solution

<Steps>
<Step title="Check browser console for errors">
Open developer tools (F12): look for failed network requests in the Network tab and JavaScript errors in the Console.
</Step>

<Step title="Hard refresh">
`Ctrl+Shift+R` (Windows/Linux) or `Cmd+Shift+R` (macOS) to clear cached assets.
</Step>

<Step title="Check database responsiveness">
Run `SELECT 1;` in Scratchpad. If even that is slow, the database connection is the bottleneck, not the UI.
</Step>

<Step title="Restart WhoDB">
```bash
docker restart whodb
```
</Step>
</Steps>

</Accordion>
</AccordionGroup>

## UI and Usability Issues

<AccordionGroup>
<Accordion title="Data not showing in table">

### Symptoms
- Table appears empty even though data exists
- After editing, changes don't appear

### Step-by-Step Solution

<Steps>
<Step title="Check if filters are applied">
WHERE conditions or the search box may be filtering out all rows. Clear filters to see all data.
</Step>

<Step title="Verify the table actually has data">
```sql
SELECT COUNT(*) FROM table_name;
```
</Step>

<Step title="Check pagination and page size">
Data might be on another page — check the page controls at the bottom.
</Step>

<Step title="Refresh">
Use the refresh action or F5 to re-fetch from the database — the frontend caches schema metadata in memory during your session, so a refresh picks up recent schema changes.
</Step>
</Steps>

</Accordion>

<Accordion title="Data editing or deletion not working">

### Symptoms
- "Edit Row", "Delete Row", or "Add Row" fails

### Step-by-Step Solution

<Steps>
<Step title="Read the error message">
- "Permission denied" → your database user lacks INSERT/UPDATE/DELETE grants
- "Constraint violation" → primary/foreign key or NOT NULL issue reported by the database
</Step>

<Step title="Test the same change in Scratchpad">
```sql
UPDATE table_name SET column = 'value' WHERE id = 123;
```

If this fails too, it's a database permissions or constraint issue — grant the needed permissions or fix the data. If it succeeds but the UI action fails, report a WhoDB issue.
</Step>
</Steps>

</Accordion>

<Accordion title="Graph visualization not showing relationships">

### Symptoms
- Graph view shows isolated tables with no edges

### Step-by-Step Solution

<Steps>
<Step title="Verify foreign keys exist">
Graph visualization is built from foreign key constraints. If the schema defines none, tables appear unconnected:

```sql
ALTER TABLE orders ADD FOREIGN KEY (user_id) REFERENCES users(id);
```
</Step>

<Step title="Refresh the graph">
Use "Refresh"/"Fit View", or a full page refresh, after schema changes.
</Step>

<Step title="Check schema selection">
Tables in other schemas or filtered out by search won't appear.
</Step>
</Steps>

</Accordion>

<Accordion title="Scratchpad cell won't execute">

### Step-by-Step Solution

<Steps>
<Step title="Check connection status">
Verify "Connected" status in the sidebar; reconnect if needed.
</Step>

<Step title="Check if another query is running">
Wait for the loading indicator to clear before executing again.
</Step>

<Step title="Focus the cell">
Click in the code editor, then press `Ctrl/Cmd+Enter` or use the Execute button.
</Step>

<Step title="Test with a trivial query">
```sql
SELECT 1;
```

If this works, the issue is your specific query; if not, it's the connection. For long-running queries that die, check database-side or proxy timeouts.
</Step>

<Step title="Create a new cell">
If one cell is stuck, add a new cell and run the query there.
</Step>
</Steps>

</Accordion>

<Accordion title="Query results look truncated">

### Step-by-Step Solution

<Steps>
<Step title="Check pagination">
Results are paginated — increase the page size or navigate pages.
</Step>

<Step title="Expand columns">
Long text is truncated for display; widen the column or click the cell to see full content in the detail view.
</Step>

<Step title="Export to verify">
CSV/Excel exports contain the full content, unaffected by display truncation.
</Step>
</Steps>

</Accordion>
</AccordionGroup>

## Finding Logs

When diagnosing server-side problems, WhoDB's logs are the first place to look:

- By default, logs go to the container/process stdout: `docker logs whodb`
- `WHODB_LOG_LEVEL` sets verbosity (`debug`, `info`, `warn`, `error`, `none`)
- `WHODB_LOG_FORMAT=json` switches to JSON logs for log collectors
- `WHODB_LOG_FILE` redirects application logs to a file (`default` means `/var/log/whodb/whodb.log`)
- `WHODB_ACCESS_LOG_FILE` redirects HTTP access logs to a file (`default` means `/var/log/whodb/whodb.access.log`)

See [Installation](/installation) for the full environment variable reference and [Monitoring Integration](/resources/integrations/monitoring) for log shipping examples.
