# Frequently Asked Questions

Find answers to the most commonly asked questions about WhoDB. If you can't find what you're looking for, check our [Troubleshooting Guide](/resources/troubleshooting) or reach out to our community.

## Installation & Setup

<AccordionGroup>
<Accordion title="How do I install WhoDB?">
WhoDB can be installed in multiple ways depending on your environment:

**Docker (Recommended)**
```bash
docker run -it -p 8080:8080 clidey/whodb
```

**Docker Compose**
```yaml
services:
  whodb:
    image: clidey/whodb
    ports:
      - "8080:8080"
```

**Binary Installation**
Download the latest binary from the [GitHub Releases](https://github.com/clidey/whodb) page and run it directly.

**Desktop Application**
Download the desktop application for Windows, macOS, or Linux from our website.

See [Installation](/installation) for the full guide, including environment variables.

<Tip>
Docker is the recommended installation method as it provides the most consistent experience across platforms
</Tip>
</Accordion>

<Accordion title="What are the system requirements for WhoDB?">
WhoDB is designed to be lightweight: 1 CPU core, 512 MB RAM, and about 50 MB of storage are enough to run it; 2 cores and 1 GB RAM are comfortable. It works on Linux, macOS, Windows, or any Docker-compatible platform, with a modern browser (Chrome 90+, Firefox 88+, Safari 14+, Edge 90+).
</Accordion>

<Accordion title="Can I run WhoDB without Docker?">
Yes, WhoDB can run as a standalone binary without Docker. Download the appropriate binary for your platform from the GitHub Releases page and execute it directly.

```bash
# Linux/macOS
./whodb

# Windows
whodb.exe
```

The binary includes everything needed to run WhoDB, including the embedded frontend.

<Info>
The standalone binary is a single file under 50MB, making it easy to deploy and manage
</Info>
</Accordion>

<Accordion title="How do I update WhoDB to the latest version?">
**Docker:**
```bash
docker pull clidey/whodb:latest
docker stop whodb
docker rm whodb
docker run -it -p 8080:8080 clidey/whodb
```

**Docker Compose:**
```bash
docker-compose pull
docker-compose up -d
```

**Binary:**
Download the latest binary from GitHub Releases and replace your existing binary.

**Desktop Application:**
Update the desktop app by downloading the latest version from [GitHub Releases](https://github.com/clidey/whodb/releases) and installing it manually.

<Warning>
Always backup your configuration and connection profiles before updating
</Warning>
</Accordion>

<Accordion title="What port does WhoDB use by default?">
WhoDB runs on port 8080 by default. You can change this by:

**Docker:**
```bash
docker run -it -p 3000:8080 clidey/whodb
```

**Environment Variable:**
```bash
PORT=3000 whodb
```

<Note>
Make sure your chosen port is not already in use by another application
</Note>
</Accordion>

<Accordion title="Can I run multiple instances of WhoDB?">
Yes, you can run multiple instances of WhoDB simultaneously by using different ports:

```bash
docker run -it -p 8080:8080 --name whodb1 clidey/whodb
docker run -it -p 8081:8080 --name whodb2 clidey/whodb
```

This is useful for connecting to different databases in separate instances, isolating environments for different teams, or testing versions side by side.

<Tip>
Each instance maintains its own connection history and settings
</Tip>
</Accordion>
</AccordionGroup>

## Database Connections

<AccordionGroup>
<Accordion title="Which databases does WhoDB support?">
WhoDB supports a broad catalog in the current CE codebase, including:

- PostgreSQL-compatible databases such as PostgreSQL, CockroachDB, and YugabyteDB
- MySQL-compatible databases such as MySQL, MariaDB, and TiDB
- SQLite3 and DuckDB
- MongoDB-compatible systems such as MongoDB, FerretDB, and DocumentDB
- Redis-compatible systems such as Redis, Valkey, Dragonfly, and ElastiCache
- ElasticSearch and OpenSearch
- ClickHouse, Memcached, and QuestDB

See our [Supported Databases](/resources/supported-databases) page for detailed version compatibility.
</Accordion>

<Accordion title="How do I connect to a database?">
Open WhoDB at http://localhost:8080, choose your database type, fill in the host, port, username, password, and database name, set SSL/TLS or advanced options if needed, and click "Connect". See [First Login](/first-login) for a walkthrough.

<Tip>
Recent connection profiles are saved locally in your browser for quick reconnection. On shared machines, clear saved connections when you're done.
</Tip>
</Accordion>

<Accordion title="Can I connect to remote databases?">
Yes, WhoDB can connect to any database accessible from your network:

**Direct Connection:**
Simply use the remote host's IP address or hostname:
```
Host: 192.168.1.100 or db.example.com
Port: 5432
```

**Private Network Or Bastion Workflow:**
If the database is only reachable through a private network, VPN, or bastion host, establish that path first. In this codebase:
- the web app expects a reachable host and port
- the CLI includes built-in SSH tunneling
- external `ssh -L` tunnels also work

**SSL/TLS:**
Enable SSL for encrypted connections:
```
SSL Mode: require/verify-ca/verify-full
SSL CA Content: (optional)
SSL Client Cert Content: (optional)
SSL Client Key Content: (optional)
```

<Warning>
Always use secure connections and private network controls when connecting to databases over public networks
</Warning>
</Accordion>

<Accordion title="How do I connect to localhost databases from Docker?">
When running WhoDB in Docker, "localhost" refers to the container itself, not your host machine. Use one of these approaches:

**On macOS/Windows:**
```bash
Host: host.docker.internal
```

**On Linux:**
```bash
docker run -it -p 8080:8080 --network host clidey/whodb
```

**Using Docker Network:**
Put WhoDB and the database on the same Docker network and use the service name as the host (for example `Host: postgres`).

<Info>
Docker networking can be complex. See our [Docker integration guide](/resources/integrations/docker) for detailed examples
</Info>
</Accordion>

<Accordion title="Can I save multiple database connections?">
Yes, WhoDB automatically saves your connection history so you can select from your recent connections when you return to the login page.

In the web app, only non-secret display details (connection name, host, database) are kept in your browser's local storage — passwords are never included, since credentials for an active login live encrypted server-side, tied to your session cookie. The desktop app stores saved connection credentials in the operating system keyring; browser-persisted state has secrets stripped on all platforms.

Features:
- Saves recent connection profiles for quick reconnect from connection history
- Edit saved connections
- Delete old connections

<Note>
Browser and app storage are local to that browser profile or machine. Use a dedicated browser profile, and log out to end the server-side session, on shared machines.
</Note>
</Accordion>

<Accordion title="What if my database requires special authentication?">
WhoDB supports several authentication methods:

**Username/Password:**
Standard authentication for most databases

**Certificate-Based Authentication:**
For databases requiring client certificates:
- Provide `SSL CA Content`
- Provide `SSL Client Cert Content`
- Provide `SSL Client Key Content`
- Optionally set `SSL Server Name`

**Cloud Provider Discovery:**
For supported AWS, Azure, and GCP integrations, enable the provider in settings and choose a discovered managed database connection.

**Environment Variables:**
For automated deployments:
```bash
export WHODB_POSTGRES_1='{"alias":"prod","host":"db.example.com","user":"readonly","password":"secret","database":"app","port":"5432"}'
```

<Tip>
Check your database provider's documentation for specific authentication requirements
</Tip>
</Accordion>

<Accordion title="How do I connect to SQLite databases?">
**Local SQLite File:**
```bash
docker run -it -p 8080:8080 -v /path/to/db:/db clidey/whodb
```

Then connect with:
```
Database Type: SQLite3
Database Path: /db/your-database.db
```

**Without Docker:**
Simply specify the full path to your SQLite file:
```
Database Path: /Users/username/data/app.db
```

<Note>
SQLite files must be accessible to the WhoDB process. Use volume mounts for Docker deployments
</Note>
</Accordion>
</AccordionGroup>

## Features & Usage

<AccordionGroup>
<Accordion title="Can I edit data directly in WhoDB?">
Yes — add rows with the "Add Row" button, edit or delete via the row's context menu, or press `Enter` / `Mod + E` on a focused row to open the edit sheet. See [Editing Records](/data/editing-records) for details.

<Warning>
All data modifications are immediate and cannot be undone. Always backup before making bulk changes
</Warning>
</Accordion>

<Accordion title="How do I import and export data?">
**Import:** WhoDB can import data files into a table with column mapping. Imports are limited to `50 MB` and `100,000` rows per file — split larger files or use database-native bulk tools. See [Importing Data](/data/importing).

**Export:** Export all rows, selected rows, or filtered/query results to CSV, Excel (.xlsx), or JSON Lines (for sources like MongoDB when available), with a configurable CSV delimiter. See [Export Options](/advanced/export-options).

<Info>
Large exports may take some time. For full backups, restores, or very large migration dumps, use your database's native tooling.
</Info>
</Accordion>

<Accordion title="What is the Scratchpad feature?">
The Scratchpad is WhoDB's query interface, similar to Jupyter notebooks for databases: multiple independent cells with syntax highlighting, query history, result views, and export. Execute with `Cmd/Ctrl + Enter`, clear the editor with `Cmd/Ctrl + U`. See [Scratchpad Introduction](/query/scratchpad-intro).

<Tip>
Use multiple cells to organize related queries and execute them independently
</Tip>
</Accordion>

<Accordion title="How does the Graph visualization work?">
The Graph view renders your schema as an interactive diagram: tables as nodes and foreign key relationships as edges, with pan, zoom, and clickable details. It works best when the database defines foreign key constraints. See [Graph View](/visualization/graph-view).
</Accordion>

<Accordion title="Can I generate test data with WhoDB?">
Yes. Open the table's context menu and choose the mock data action (or press `Cmd/Ctrl + Shift + G` while the table has focus), set the row count and Append/Overwrite mode, and generate. Values are auto-generated based on column types. See [Mock Data](/advanced/mock-data).

<Warning>
Overwrite mode will delete all existing data in the table before inserting mock data
</Warning>
</Accordion>

<Accordion title="Does WhoDB support transactions?">
Single-row UI operations execute immediately in auto-commit mode. For multi-step operations that should succeed or fail together, use explicit transaction commands in the Scratchpad:

```sql
BEGIN;
UPDATE users SET status = 'active' WHERE created_at < '2024-01-01';
UPDATE orders SET processed = true WHERE user_id IN (SELECT id FROM users WHERE status = 'active');
COMMIT;
```
</Accordion>

<Accordion title="Can I use WhoDB with multiple schemas?">
Yes. Select the schema from the sidebar (PostgreSQL), switch between databases (MySQL, MongoDB), and use qualified names for cross-schema queries. Schema support varies by database type — see [Supported Databases](/resources/supported-databases).
</Accordion>

<Accordion title="How do I search for specific data?">
Use the quick search box for real-time filtering across visible columns, WHERE conditions for operator-based filters (`=`, `!=`, `>`, `LIKE`, `IN`, combined with AND/OR), or the Scratchpad for complex queries. See [Filtering and Searching](/data/filtering-searching).
</Accordion>
</AccordionGroup>

## AI & Chat Features

<AccordionGroup>
<Accordion title="How do I enable the AI chat feature?">
WhoDB integrates with multiple AI providers:

**Ollama (Local):**
```bash
# Install Ollama from https://ollama.com
ollama pull llama3.1
```
WhoDB auto-detects Ollama on localhost:11434

**OpenAI:**
```bash
docker run -e WHODB_OPENAI_API_KEY=your-key clidey/whodb
```

**Anthropic:**
```bash
docker run -e WHODB_ANTHROPIC_API_KEY=your-key clidey/whodb
```

Once configured, the Chat feature appears in the sidebar.

<Info>
Ollama runs entirely on your machine, keeping your data private. Cloud providers (OpenAI/Anthropic) send queries to their APIs
</Info>
</Accordion>

<Accordion title="What can I ask the AI assistant?">
The AI assistant can help with:

**Query Generation:**
- "Show me all users created in the last 30 days"
- "Find the top 10 products by sales"

**Data Analysis:**
- "What's the distribution of user ages?"
- "Summarize the orders table"

**Schema Understanding:**
- "What tables are related to users?"
- "Explain the relationship between orders and customers"

<Tip>
The AI has full context of your database schema, so ask natural questions without specifying table names
</Tip>
</Accordion>

<Accordion title="Is my data sent to AI providers?">
**Ollama (Local):**
- Runs entirely on your machine
- No data leaves your computer
- Complete privacy

**OpenAI/Anthropic:**
- Query text and schema information are sent to their APIs
- Actual data values are not sent unless explicitly mentioned in your question
- Protected by API provider's privacy policies

**What's Sent:**
- Table and column names
- Data types and relationships
- Your question text
- Generated SQL queries

**What's NOT Sent:**
- Actual row data
- Connection credentials
- User passwords
- Database contents

<Warning>
If you're working with sensitive data, use Ollama for complete privacy
</Warning>
</Accordion>

<Accordion title="Can the AI modify my data?">
Yes, with a confirmation step for anything that changes data:

- AI-generated SELECT queries run automatically and show their results in the chat
- Modification and DDL statements (INSERT, UPDATE, DELETE, CREATE, ALTER, DROP) are **not** executed immediately — the chat pauses with an inline confirmation showing the generated SQL, with Confirm and Cancel buttons
- Clicking Confirm executes the statement; clicking Cancel discards it

<Note>
This confirmation step prevents accidental data loss or corruption. Review the SQL before confirming
</Note>
</Accordion>
</AccordionGroup>

## Security & Privacy

<AccordionGroup>
<Accordion title="Is WhoDB secure?">
WhoDB implements multiple security measures:

**Connection Security:**
- SSL/TLS support for database connections
- Certificate-based authentication
- Reverse-proxy and network-based deployment controls

**Application Security:**
- Open-source code for review
- Web app login credentials are encrypted and stored server-side, never in browser storage; only non-secret connection metadata is kept in the browser
- Environment-defined profiles keep connection details in server-side environment configuration

**Deployment Security:**
- Docker security best practices
- Minimal attack surface
- Regular security updates
- Dependency scanning

<Info>
WhoDB is open source, allowing security audits by the community
</Info>
</Accordion>

<Accordion title="Where are my passwords stored?">
WhoDB follows these security principles:

**Connection Passwords:**
- Web app: passwords never reach browser storage. A successful login is encrypted (AES-256-GCM) and stored server-side in a local session database; the browser only holds an opaque, `HttpOnly` session cookie
- Desktop app: stores saved connection credentials in the operating system keyring, falling back to local config only if no keyring is available; browser-persisted state has secrets stripped on all platforms
- Environment-defined profiles keep credentials in server-side environment configuration
- Desktop credential security ultimately depends on the operating system account protecting that machine

**Re-authentication:**
- Log out to delete the server-side session immediately (web app), or clear saved connections on the desktop app
- Prefer read-only accounts and dedicated credentials for operational access

<Warning>
Never commit connection strings with passwords to version control or share them in screenshots
</Warning>
</Accordion>

<Accordion title="Can I use WhoDB in production?">
WhoDB can be used in production with proper precautions:

**Recommended Practices:**
- Use read-only database accounts when possible
- Deploy behind authentication at your reverse proxy or ingress
- Limit network access with firewalls
- Enable SSL/TLS for all connections
- Regular security updates
- Database-native audit logs for sensitive environments

**Use Cases:**
- Development and staging environments
- Read-only production access for support teams
- Database administration with proper access controls
- Data analysis and reporting

<Warning>
Never expose WhoDB directly to the public internet without authentication and access controls
</Warning>
</Accordion>

<Accordion title="How do I secure my WhoDB deployment?">
**Reverse Proxy with Authentication:**
```nginx
location /whodb {
    auth_basic "WhoDB Access";
    auth_basic_user_file /etc/nginx/.htpasswd;
    proxy_pass http://localhost:8080;
}
```

**Firewall Rules:**
```bash
# Only allow specific IPs
iptables -A INPUT -p tcp --dport 8080 -s 10.0.0.0/8 -j ACCEPT
iptables -A INPUT -p tcp --dport 8080 -j DROP
```

**Container Security:**
```bash
# Run as non-root user
docker run --user 1000:1000 clidey/whodb
```

<Note>
The image writes session data to the `/data` volume. When running as a non-root user, make sure the volume is owned by (or writable for) that UID, or sessions will fail.
</Note>

<Tip>
Combine multiple security layers for defense in depth
</Tip>
</Accordion>

<Accordion title="Does WhoDB log my queries?">
WhoDB logging behavior:

**Application Logs:**
- Connection attempts (without passwords)
- Error messages
- Performance metrics
- Feature usage (anonymous)

**Query Logs:**
- Not logged by WhoDB server
- Saved in browser history (client-side only)
- Can be cleared by user
- Never transmitted to external servers

**Database Logs:**
- Your database may log queries independently
- Check your database's logging configuration
- Not controlled by WhoDB

<Note>
Browser history is local to your machine and not shared with WhoDB servers
</Note>
</Accordion>
</AccordionGroup>

## Performance & Optimization

<AccordionGroup>
<Accordion title="How does WhoDB handle large datasets?">
WhoDB paginates automatically with a configurable page size, virtualizes table rendering, and adds LIMIT clauses to keep result sets bounded. For tables with millions of rows, always use WHERE conditions and LIMIT. See the [Performance Tuning Guide](/resources/performance-tuning).
</Accordion>

<Accordion title="Can I increase connection limits?">
The current CE web app manages an internal SQL connection pool with fixed defaults:
- Max open connections: `10`
- Max idle connections: `5`
- Max connection lifetime: `30 minutes`
- Max idle lifetime: `5 minutes`

If you need more concurrency, tune the database server itself or add a database-side pool such as PgBouncer or ProxySQL.

<Warning>
Be careful not to exceed your database server's connection limits
</Warning>
</Accordion>

<Accordion title="Does WhoDB cache query results?">
The frontend caches GraphQL responses (such as table lists and column metadata) in memory for the duration of your session, so repeat navigation doesn't re-query the database. Refreshing the page or using the refresh action re-fetches from the database.

Cloud-provider connection discovery results are cached server-side and refresh after 5 minutes.

<Info>
If the schema changed and the UI looks stale, refresh to re-fetch
</Info>
</Accordion>
</AccordionGroup>

## Migration & Integration

<AccordionGroup>
<Accordion title="Can I migrate from other database tools?">
Yes — recreate your connections in WhoDB with the same host, port, database, and user, and copy important SQL into Scratchpad pages or your repository. Dedicated guides: [migrating from phpMyAdmin](/guides/migrating-from-phpmyadmin) and [migrating from pgAdmin](/guides/migrating-from-pgadmin).

<Tip>
WhoDB can run alongside existing tools during your transition period
</Tip>
</Accordion>

<Accordion title="Can I integrate WhoDB with my CI/CD pipeline?">
Yes. Use the web container when you want a temporary inspection UI, and use `whodb-cli` or native database clients for scriptable pipeline steps.

**WhoDB web service:**
```yaml
# GitHub Actions example
- name: Start WhoDB for inspection
  run: |
    docker run -d -p 8080:8080 clidey/whodb:latest
    curl -fsS http://localhost:8080/health
```

**Scriptable checks:**
```bash
whodb-cli query "SELECT COUNT(*) FROM users" --connection mydb --format json
whodb-cli export --connection mydb --table users --output users.csv
```

Use native database clients for migrations, full backups, restores, and SQL seed files. See our [CI/CD Integration Guide](/resources/integrations/ci-cd) for detailed examples.
</Accordion>

<Accordion title="How do I backup data using WhoDB?">
Export table data as CSV or Excel from the table view, or script it:

```bash
whodb-cli export --connection mydb --table users --output users.csv
```

For schema DDL and large-scale backups, use native database tools (`pg_dump --schema-only`, `mysqldump --no-data`, and so on).

<Warning>
WhoDB exports are suitable for small to medium datasets. Use database-native tools for large-scale backups
</Warning>
</Accordion>

<Accordion title="Can I use WhoDB with monitoring tools?">
Yes. Monitor WhoDB with its health endpoint, container/runtime metrics, database exporters, and logs.

**Health check:**
```bash
curl -fsS http://localhost:8080/health
```

**Logging Integration:**
```yaml
environment:
  - WHODB_LOG_LEVEL=info
  - WHODB_LOG_FORMAT=json
  - WHODB_LOG_FILE=default
  - WHODB_ACCESS_LOG_FILE=default
```

See our [Monitoring Integration Guide](/resources/integrations/monitoring) for detailed setup.

<Note>
Health check endpoints are useful for load balancer configuration
</Note>
</Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
<Accordion title="I can't connect to my database. What should I check?">
**Checklist:**
1. Verify connection details (host, port, username, password)
2. Check the database is running and reachable from where WhoDB runs (`nc -zv host port`)
3. Check SSL/TLS requirements and the `SSL Mode` setting
4. Docker: use `host.docker.internal` instead of `localhost` for host databases
5. Verify the database user may connect from WhoDB's address

See our [Troubleshooting Guide](/resources/troubleshooting) for detailed solutions.
</Accordion>

<Accordion title="WhoDB is running but I can't access the web interface">
**Checklist:**
1. Verify WhoDB is running: `docker ps` or process check
2. Check port mapping is correct and try http://localhost:8080
3. Verify no other service is using port 8080: `lsof -i :8080`
4. Check browser console for errors

```bash
# Use a different port if 8080 is taken
docker run -p 3000:8080 clidey/whodb
```
</Accordion>
